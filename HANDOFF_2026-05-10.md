## TL;DR — what changed today

### Architectural

1. **Two-channel architecture** — `goal:N` / `heartbeat:N` /
   `sunday_test_*` proliferation collapsed into TWO real conversations:
   - `JOHN_CHARLES` (`8455750177`) — the relational thread (UI/Telegram/iMessage).
   - `CHARLES_LOG` (`charles_log`) — operational/autonomous stream.
2. **JOHN_CHARLES is read-only-clean** — only user + final assistant
   rows persist there. No tool plumbing, no "let me X" intermediate
   reasoning. CHARLES_LOG keeps the full record. Cemented in
   `feedback_john_charles_clean_dialog.md` and as a docstring invariant
   in `core/agent.py`.
3. **POST routing pinned** — `/api/command/message` always routes to
   JOHN_CHARLES regardless of which conv the UI sidebar selected.
4. **Round caps differ** — JOHN_CHARLES = 5 rounds (`MAX_TOOL_ROUNDS_RELATIONAL`),
   CHARLES_LOG = 25.
5. **Cross-channel context** — when John speaks, Charles's prompt gets
   a "Background — reference only, answer John, don't continue this work"
   summary of recent CHARLES_LOG activity. Critical framing.
6. **John-vocab tag index** — every auto-remembered fact gets `john:<kw>`
   tags from John's recent messages so his blue-collar phrasing pulls
   the fact via auto-recall.

### New tools / behaviors

7. **`vibe_check` tool** — returns one-line italic mood reflecting state.
8. **Welcome-back greeting** — first reply after >6h gap from John gets

9. **Boss Hog inbound iMessage** — watchdog polls `chat.db` every tick,

## Cement points (for future-Claude reading this)

These are durable invariants — break them and John's chat goes back to
the bloat-and-loop state:

1. `JOHN_CHARLES = '8455750177'` and `CHARLES_LOG = 'charles_log'`. Period.
2. POST `/api/command/message` ALWAYS uses JOHN_CHARLES, never the
   client-supplied conv_id.
3. `memory.log_assistant_tool_calls` and `memory.log_tool_result` are
   GUARDED by `if conversation_id != channels.JOHN_CHARLES` in
   `core/agent._respond_impl`. Adding a third logging path? Same guard.
4. `MAX_TOOL_ROUNDS_RELATIONAL = 5` for JOHN_CHARLES. Don't raise this —
   if Charles needs more rounds for a JOHN_CHARLES message, the model
   is misinterpreting the message, not actually solving the task.
5. `_build_charles_log_summary` framing must include "do NOT continue
   this work in this reply." Without that line, Charles picks up the
   bullets as work-to-do and burns 20 rounds.
6. New auto-saving fact paths should call `_with_john_vocab_tags(base_tags)`
   so the John-vocab index keeps growing.
